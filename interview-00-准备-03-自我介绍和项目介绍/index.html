项目介绍

数据安全管理平台

数据安全管理平台主要是为企业提供一站式的安全管理的服务和能力，包括：数据源管理、敏感数据发现、数据分类分级、数据动态脱敏和静态脱敏、数据库水印和溯源，然后还接入了一些其他部门的能力 数据库审计和数据库防火墙。

  • 数据源管理：接入了众多数据库，如 MySQL、Oracle、PostgreSQL、SqlServer、hive、MongoDB、达梦、瀚高等等。然后获取数据源 元信息。
  • 敏感数据发现。
    • 系统内置了一些发现规则，包括 正则、字典。可以匹配 字段名、字段值和字段注释。还有排除的功能。用户可以自定义规则，然后还可以设置 优先级。
    • 用户需要创建一个敏感数据发现的模板、选择一些规则，然后选择抽样的条数以及抽样的比例，并选择匹配的比例。
      其实之前有设计说按照 有无 唯一索引或者主键索引以及自增来设计抽样。对于有的数据库，不设置抽样条数，只用设置抽样的比例。对于没有的数据库，必须要设置抽样的条数。
    • 然后用户创建一个敏感数据发现任务，选择数据源和模板、及扫描的时间。系统会自动的在规定的时间进行扫描。
  • 数据分类分级。
    • 用户先创建分类、然后再创建规则的时候关联分类。
    • 扫描完成之后，会形成一个初步的分类分级，然后用户可以上去进行调整，调整完之后，点确定，然后就形成了分类分级。
  • 静态脱敏
    • 用户先创建脱敏的规则，比如说哪个类型对应哪个脱敏规则，规则有遮掩，就是用 * 替代、有固定替换、随机替换、1:1 替换、加密 和 md5、以及擦除。
    • 然后创建脱敏的模板，选择对应的脱敏规则
    • 然后创建脱敏任务，选择脱敏的源和目标，脱敏的模板
  • 动态脱敏
    • 动态脱敏我们现在做的是 提供一个api，后端连接到对应的数据库表，用户直接访问 api 的方式来获取后端的数据，我们之前调研了，动态脱敏有三种形式，一个是用户输入一个数据，返回一个脱敏的数据，一个是我们现在的形式，一个是类型数据库防火墙的形式。我做调研的时候和老大说了这三种形式，也说了每个形式的优缺点，但是我们人手不够，而且没有这方面的技术栈，所以第三种新式搞不了。
    • 我们的方法是选择一个 api，用户选择后面对应的数据库表、查询的字段、脱敏的字段、和其他表的连接等等
  • 数据库水印和溯源
    • 水印分为三种：无痕水印和伪行水印、伪列水印
    • 用户先创建水印值，然后选择数据源、目标地址、水印方式。
  • 数据库审计
    • 有一个终端软件，需要安装到对应的服务器上，然后抓包，分析
  • 数据库防火墙
    • 创建数据源，然后开启一个服务，所有的请求都通过这个请求转发到后端的防火墙
    • 主要是能防一些 SQL 注入，还有后端 数据库的 漏洞。

安全数据沙箱

安全数据沙箱是一个为企业提供安全的数据分析环境的一个系统。简单来说，就是将数据分析过程和真实的数据隔离开来，在进行分析建模的过程中，分析人员是看不到真实数据的，只有实际运行的时候才能看到真实的数据。

整个系统分为两个环境，一个是运行环境，一个是调试环境。

在运行环境，主要的功能包括

  • 多种数据源接入，如 MySQL、Redis、MongoDB、SqlServer、Oracle、达梦等等数据库，然后再这个基础上抽象出来一个数据集，因为数据源的数据不一定都需要，所以抽象出来数据集。

  • 对数据集进行抽样和脱敏之后就会形成样本数据。在调试环境用户看到的就是这个数据。

    • 为了更好的脱敏和复用，我们把这里做成了一个单独的模块，叫做 DLP，这个模块分为几部分：
    • dlp-service：负责接受请求，根据请求参数进行数据处理。参数分为 结构化、非机构化、以及图片。
    • ocr：如果数据为图片，需要进行 ocr 识别，这个是我搞的，用的 yolo5，做的，框架的 paddlepaddle。
    • nlp：这个是别人搞得。
    • desensitization：具体实现脱敏的模块。
    • 模块间使用 gRPC 调用，后端程序先构造请求参数，发送到 dlp service，service 根据参数的不同调用不同的模块。

  • 实验管理。可以把实验理解为一个比赛，所有的用户都可以报名这个实验。在运行环境创建实验，并为实验添加数据集，添加的数据集需要进行审批，如果审批通过，会将上面抽样和脱敏形成的数据上送到调试环境，用户就能看到这些数据。

调试环境

数据分析师的工作环境主要在调试环境。分析师参加一个实验，然后可以创建一个分析环境。分析环境的页面是一个 dagrejs，在页面可以选择数据源，可以创建一个 Jupyter 分析环境、还可以选择输出数据源，在 Jupyter 里面进行具体的分析工作，分析师可以建模。

具体的执行流程是：

用户选择数据源 –> 开启 Jupyter 进行建模 –> 选择输出的数据源

后台会根据 用户的流程，自动的调用 kubeflow 进行整个分析的流程。

  • 页面上会根据选择的数据源自动生成一个样本文件，传入到 HDFS 上。Jupyter 获取数据是通过我们封装的一个库，这个库需要指定文件地址和文件类型，调用接口之后，自动的从文件中加载数据并返回。用户需要按照 包的格式输出，然后会把输出文件上传到 HDFS 中。用户在界面上配置相应的输出之后，会把相应的文件转化成相应的输出。
  • 运行全量时，使用 kubeflow 来对这些容器进行编排。
    • 获取数据源，把数据源输出为文件。
    • 运行 jupyter 代码，代码获取全量数据，进行训练，并输出文件。
    • 输出节点，把文件输出到 HDFS 上。

点击运行全量之后，会输出文件到 HDFS 上，用户提交申请，申请通过之后可以下载。

VMS 漏洞管理系统

漏洞管理系统，是一个以资产为中心的，对不同来源的漏洞进行全生命周期管理的一个系统。

首先，VMS 是以资产为中心的，所以 VMS 有资产管理的功能。

其次，VMS 是对不同来源的漏洞进行管理的一个系统，不同来源说的是，旷视的漏洞来源共分以下几个，第一个是提测发现的漏洞，第二是外部白帽子（白帽子可以理解为公司外部的做渗透的人）发现的漏洞，第三个是众测发现的漏洞，分别对这三类漏洞进行管理。

全生命周期管理是指，漏洞管理实现了闭环，比如提测的漏洞，有提测，排期，测试，修复，复测，结束几个阶段，其他来源的漏洞的管理和这个思路是一样的。

这是这个系统主要的功能，除此之外还有一些其他的功能：

  • 比如和 Jira 打通，因为业务测很多项目使用 Jira 对项目进行管理，VMS 的漏洞会自动同步到 Jira 上，如果 Jira 上修复了，结果也会自动同步到 VMS 上
  • 和火线平台打通，VMS 会自动去拉取 火线平台的漏洞数据。
  • 排期功能，漏洞可以选择排期，到时间维修会自动提醒。
  • 统计功能，从多维度对漏洞进行统计。比如部门维度，公司维度，修复未修复维度，漏洞类型维度。
  • 我们这个系统对接了 ASTE 系统，提供自动化扫描能力。

ASTE 自动化安全测试引擎

先说一下背景，就是之前某一段时间，安全提测太多了，安全同学测试不过来，说能不能开发一个这种安全测试的工具，可以减轻测试同学的压力。

正好也收到了来自产线开发/测试同学的反馈，希望安全部的测试能力对外输出并能为其赋能，使其具备简单的安全测试能力，从而在开发生命周期更早的阶段发现安全问题，增加安全问题发现率，减少解决安全问题的成本。

正是出于这个目的，我们开发了 ASTE，简单来说，可以把 ASTE 理解为一个集合很多自研的已经开源的扫描工具的集合，用户只需要规范，输入特定格式的参数，比如 web 地址，git 地址，ASTE 就会自动调用后端的工具进行扫描，并输出返回结果。

目前我们把 ASTE 拆成了很多的模块，每个模块负责不同的功能，所有的模块都是容器化部署，实现了高可用和后端扫描工具的动态扩容，以及动态增加不同类型的扫描工具，比如现在接入了 xray，你想接入 awvs，是直接可以接入的。而不需要重启。

ASTE 实现了服务降级的功能，还实现了统计扫描任务的功能，包括历史扫描任务和正在扫描的扫描任务。

1、前端交互部分：主要负责接受扫描任务，并根据扫描参数将扫描分发到不同的扫描工具，提供前端访问，可以通过前端页面发送扫描任务和实时查看所有任务的扫描状态以及扫描结果，也可以通过前端页面进行简单的配置修改。

2、工具脚本部分：主要负责接受从核心引擎分发的扫描任务，并负责和具体的工具进行交互以完成本次扫描，最后将扫描结果返回到扫描引擎。

3、工具：具体的扫描工具。

4、数据处理部分：负责汇总扫描结果，分析扫描状态，确保扫描状态正确，并将扫描结果写入数据库。

  • 端口扫描：nmap
  • ssl 扫描：ssl
  • 目录扫描：directory_scan
  • 指纹识别：finger
  • 依赖检测：
    • dependency_track：
    • trivy：简单地说，就是扫描应用的容器镜像，找一找有没有已知的安全漏洞。GitLab 认为其比 Clair 更易于维护，去年开始用 Trivy 代替了后者。
  • 黑盒扫描：zap、xray（这个有代理模式）、awvs、AppScan（这两个有插桩模式）。
  • 白盒扫描：cobra

DSP 数据安全管控平台

数据安全管控平台，简单来说就是一个做数据管理的平台。

首先来说，DSP 是一个安全治理的项目，平台是治理的产物。

先说一下背景，我们之前对旷视的信息资产进行过梳理，发现旷视的主要资产存储在几个地方，一个是 wiki，git 这种共享的平台上面，还有一个旷视最重要的资产存储在一个神秘的地方。

这个地方有很多的存储桶，我们把他叫做 bucket，好做资产都是以 bucket 为单位进行管理的，我们发现安全部对这些桶没有数据安全的管理。所以我们就想把这些 bucket 管理起来。

dsp 主要做了四个模块，第一个模块是把所有的 bucket 分级，第二个模块是提供 bucket 密级的修改功能，第三个模块是提供 bucket 下载申请功能，第四个模块是提供 bucket 下载审计功能。

每一个模块都有更细的逻辑，

比如第一步就是分类分级，我们梳理了 bucket 的使用者，发现共有三类使用者，第一类是个人，第二类是组账号，第三类是各种平台，如蜂巢，月影，众智， hubble。

然后分别针对每类使用者，指定密级标记的方式，比如个人使用者，我们刚开始是把数据导出来，给每个人，让他们标记之后，反馈回来，有了平台之后，直接使用平台发送钉钉，提醒他们上来标记。

针对组账号的这类bucket，我们是找组账号的管理员，默认让管理员来标记bucket。

针对第三方系统调用创建的bucket， 先把 这类bucket 进行分类整理，然后按照分好的类去和对应的系统负责人对，看每一类型的bucket 的密级怎么标，密级是什么，然后把这个规则 转变成代码，使用 dsp 后台程序去跑。

对于bucket 是不断创建的，所以这个系统处理两种类型的bucket，一种是存量的，一种是增量的，存量和增量的都会运用这个规则。

所有的bucket都是从 mongodb 数据库获取的，还会有一些增量的数据，为了环境数据库的压力，我们使用了全量同步和增量同步的方式。

dsp 除了标密级的功能之外，还增加了下载审批的功能，如果你想下载到外部平台，需要走dsp 的数据审批功能。

如果没有审批就下载了，dsp 还做了审计的功能，dsp 和 splunk 联动。

SEM 安全事件管理

安全部有很多的事件，比如，高危的 cve 漏洞、wiki 异常下载、git 异常下载、bucket 异常下载，在没有这个平台以前，都是人工手动去通知每个人，手动处理，很不方便，而且统计也比较麻烦。

所以就想做专注于事件管理的平台，自动化的去发送一些告警，并提供告警处理的功能，如果规定时间内没有处理告警，还会重复发送告警，重复发送还未处理，就会升级 leader 告警。

这个平台是常用的这种前后端的架构，前端 vue，后端 flask + gunicron

这个平台有两个好的设计，第一个是可以随时添加事件，不用重启，而且配置这个事件前端的展示，如 图标，事件名，事件详情，事件的操作，都是可以配置的。所有的数据都是后端动态生成的，包括vue 的路由，每两秒刷新一次。

这样新增一个事件的成本就变的很低。

第二个，每个事件发送本身都可以进行详细的配置，这个也是和业务同学详细沟通之后总会设计的，如是否过滤，发送类型，都是可以动态配置的。

目前已经发送了 4000+ 告警了吧。

风控平台

顾名思义，他就是一个风控平台，和常规的风控平台一样，对业务的日志进行分析，发现其中的 异常行为。

我们开发的风控平台有三个部分构成：

  • Flink。flink 是业内成熟的流式数据处理框架，效率高，用于对无界和有界数据流进行有状态计算。所以我们选择 Flink 对数据进行分析。

  • notify。负责从 redis 中获取数据，并分析和处理数据并落库，负责发送告警

  • alert。是一个前后端的平台，负责展示 发现的告警，并提供交互，处理告警。

MySQL 数据安全项目

先说一下背景，泰康的后端服务的数据库，很多用的是 MySQL，但是存在 MySQL 明文存储敏感信息的现象，这个肯定是不合规的，所以当时我们就做一下 MySQL 的安全治理，所以就有了这个项目。

这个项目第一步就是做数据库的管理，需要把数据库的一些信息录入到系统里面，比如 IP，PORT，账号，密码等。账号肯定是只读账号，密码在存储的时候也是加密存储的。

有了账号之后，下一步是扫描，扫描提供两种方式，一种是马上进行扫描，一种是可以设置定时任务进行扫描。

所有的扫描请求都会发到 celery 中，并且我们在5台服务器上同时启动了worker 负责接收celery的扫描请求。

在扫描的过程中，扫描的状态数据也会实时同步到 redis 中，扫描完成的结果直接持久化到数据库中。

在前端页面会对当前的扫描进行统计。你也可以通过查看数据的扫描历史查看这个数据库在什么时候进行了扫描，并且可以点击详情，查看是否有敏感数据。

加密方式有DES、3DES、AES（推荐）

随机森林的优势：

表现性能好，与其他算法相比有着很大优势。
随机森林能处理很高维度的数据（也就是很多特征的数据），并且不用做特征选择。
在训练完之后，随机森林能给出哪些特征比较重要。
训练速度快，容易做成并行化方法(训练时，树与树之间是相互独立的)。
在训练过程中，能够检测到feature之间的影响。
对于不平衡数据集来说，随机森林可以平衡误差。当存在分类不平衡的情况时，随机森林能提供平衡数据集误差的有效方法。
如果有很大一部分的特征遗失，用RF算法仍然可以维持准确度。
随机森林算法有很强的抗干扰能力（具体体现在6,7点）。所以当数据存在大量的数据缺失，用RF也是不错的。
随机森林抗过拟合能力比较强（虽然理论上说随机森林不会产生过拟合现象，但是在现实中噪声是不能忽略的，增加树虽然能够减小过拟合，但没有办法完全消除过拟合，无论怎么增加树都不行，再说树的数目也不可能无限增加的。）
随机森林能够解决分类与回归两种类型的问题，并在这两方面都有相当好的估计表现。（虽然RF能做回归问题，但通常都用RF来解决分类问题）。
在创建随机森林时候，对generlization error(泛化误差)使用的是无偏估计模型，泛化能力强。

应用防护系统（也称为：网站应用级入侵防御系统。英文：Web Application Firewall，简称：WAF）。利用国际上公认的一种说法：Web应用防火墙是通过执行一系列针对HTTP/HTTPS的安全策略来专门为Web应用提供保护的一款产品。

WAF可以实现如下功能

a、防止SQL注入、本地包含、部分溢出、Fuzzing测试、XSS等Web Attack；

b、防止SVN/备份之类的文件泄漏；

c、防止Apache Bench之类的压测工具Attack；

d、屏蔽常见的Hacker扫描工具；

e、屏蔽异常的网络请求；

f、屏蔽图片附件类目录的PHP执行权限；

g、防止Webshell上传等。

WAF与网络防火墙的区别

网络防火墙作为访问控制设备，主要工作在OSI模型三、四层，基于IP报文进行检测。只是对端口做限制，对TCP协议做封堵。其产品设计无需理解HTTP会话，也就决定了无法理解Web应用程序语言如HTML、SQL语言。因此，它不可能对HTTP通讯进行输入验证或攻击规则分析。针对Web网站的恶意攻击绝大部分都将封装为HTTP请求，从80或443端口顺利通过防火墙检测。

一些定位比较综合、提供丰富功能的防火墙，也具备一定程度的应用层防御能力，如能根据TCP会话异常性及攻击特征阻止网络层的攻击，通过IP分拆和组合也能判断是否有攻击隐藏在多个数据包中，但从根本上说他仍然无法理解HTTP会话，难以应对如SQL注入、跨站脚本、cookie窃取、网页篡改等应用层攻击。

web应用防火墙能在应用层理解分析HTTP会话，因此能有效的防止各类应用层攻击，同时他向下兼容，具备网络防火墙的功能。

WAF支持HTTPS业务，并且支持泛域名接入。只需根据提示将SSL证书及私钥上传，WAF即可防护HTTPS业务流量。配置HTTPS业务接入后，WAF会先解密访问请求，检查请求包，再重新加密，并转发正常的请求到源站。

敏捷开发

敏捷开发是以人为中心，快速开发、迭代、循序渐进的开发方式。

在敏捷开发中，软件项目的构建被切分成多个子项目，各个子项目的成果都经过测试，具备集成和可运行的特征。

更加单的说，敏捷开发并不是追求前期完美的设计、完美编码，而是力求在很短的周期内开发出产品核心功能，尽早发布出可用的版本。然后在后续的生产周期内，按照新需求不断迭代升级，完善产品。

  1. 我们首先需要确定一个产品需求列表（Product Backlog），这个是由PO负责的。

  2. 有了Product Backlog，我们需要通过Sprint Planning Meeting（Sprint计划会议）来从中挑选出一个Story作为本次迭代完成的目标。

  3. Sprint Backlog是由Scrum Team完成的，每个成员根据Sprint Backlog再细化成更小的任务（每个任务的工作量可以在2天内完成）。

  4. 进行Daily Scrum Meeting（每日站会），每次会议控制在10分钟以内，每个人都必须发言，并且要向所在组成员当面汇报你昨天完成了什么，并且向所有组成员承诺你今天要完成什么，同时遇到不能解决的问题也可以提出，相关人员在会后进行解答、处理。

  5. 当一个Story完成，也就是Sprint Backlog被完成，也就表示一次Sprint完成，这时，我们要进行Sprint Review Meeting（演示会议），也称为评审会议，产品负责人和客户都要参加（最好本公司老板也参加），每一个Scrum Team的成员都要向他们演示自己完成的软件产品。

  6. 最后就是Sprint Retrospective Meeting（回顾会议），也称为总结会议，以轮流发言式进行，每个人都要发言，总结讨论改进的地方，放入下一轮Sprint的产品需求中。

总结：1、确定产品需求列表-2、Sprint计划会议-3、Strum Team开始分工Sprint-4、Daily Strum Meeting每日站会-5、Sprint Review Meeting，Sprint的评审会议，演示会议-6、Sprint Retrospective Meeting，Sprint的总结会议。

一些想法

其实可以看到，怎么设计高并发系统这个问题本身他是不难的，无非是基于你知道的知识点，从物理硬件层面到软件的架构、代码层面的优化，使用什么中间件来不断提高系统的抗压能力。但是这个问题本身会带来更多的问题，

微服务本身的拆分带来了分布式事务的问题，http、RPC框架的使用带来了通信效率、路由、容错的问题。
MQ的引入带来了消息丢失、积压、事务消息、顺序消息的问题。
缓存的引入又会带来一致性、雪崩、击穿的问题。
数据库的读写分离、分库分表又会带来主从同步延迟、分布式ID、事务一致性的问题。
而为了解决这些问题我们又要不断的加入各种措施熔断、限流、降级、离线核对、预案处理等等来防止和追溯这些问题。

  2. 公司对这个岗位的期待和定位是什么。

设计系统

业务架构，简单的说就是从业务的角度出发，或者说从使用者的角度出发，看他们能从系统中得到什么。“企业希望从这个产品身上得到什么”，以及“用户想从产品身上得到什么”。

为了实现这一目标，我们还必须弄清楚那些用户，能够通过什么渠道达成他们的目标，甚至还可能会有其他的目标。

也就意味着产品端必须清晰的规划出一个完整的业务互动的过程，搞清楚一个产品涉及到的各种不同的角色，他们所期望的目标，以及他们的需要完成的业务动作，从而为产品设计打下基础。

产品架构

按照产品架构图的结构和路径，产品的RoadMap也就可以被清晰的拆解出来，同时它还能指导技术架构的选型，比如业务的容量，扩展性等等，为未来的产品发展指明了方向；

产品边界：不做什么，很多时候更为重要，明确一个产品的基本边界，可以让这个产品少走很多弯路，降低很多的风险和不必要的成本。这对于创业团队来说，有些时候非常关键。

系统架构可以分为两部分，一部分是 技术选型，一部分是 把系统分成不同的模块。

技术架构解决的问题包括：如何进行纯技术层面的分层、开发框架的选择、开发语言的选择、涉及非功能性需求的技术选择。由于应用架构体系是分层的，那么对于的技术架构体系自然也是分层的。大的分层有微服务架构分层模型，小的分层则是单个应用的技术分层框架。大的技术体系考虑清楚后，剩下的问题就是根据实际业务场景来选择具体的技术点。各个技术点的分析、方案选择，最终形成关键技术清单，关键技术清单考虑应用架构本身的分层逻辑，最终形成一个完成的技术架构图。

按照解决的业务问题域，划分出不同的功能模块，再根据功能模块间的关系，组合成子系统。应用架构在产品架构的基础上考虑两个事情：第一、考虑的是子系统间的关系。第二、考虑将可复用的组件或模块进行下沉，沉淀到平台层，为业务组件提供统一的支撑。
